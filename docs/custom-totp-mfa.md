# Envault-managed authenticator MFA

Envault uses Firebase Authentication for the email/password first factor, but it
does not depend on Firebase Identity Platform MFA.

The second factor uses standard RFC 6238 TOTP codes and is compatible with
Microsoft Authenticator, Google Authenticator, Authy and similar applications.

## Server configuration

Generate a dedicated 32-byte encryption key:

```bash
openssl rand -base64 32
```

Configure the resulting value as:

```dotenv
MFA_ENCRYPTION_KEY=generated-base64-value
```

Set the same key in local development and every deployed server environment.
Never expose it through a `NEXT_PUBLIC_` variable, commit it to Git, log it, or
change it while users have MFA enrolled. Losing or rotating this key without a
migration makes existing TOTP secrets unreadable.

## Security model

- TOTP secrets are generated by Envault.
- Only the standard `otpauth://` URI and setup secret are shown during
  enrollment.
- Stored secrets are encrypted with AES-256-GCM.
- Verification occurs only on the server.
- A Firebase ID token is not exchanged for an Envault session until TOTP
  verification succeeds.
- The same TOTP time-step code cannot be replayed.
- Removing MFA requires a current authenticator code.
- Missing encryption keys and unavailable MFA storage fail closed.

## Free-tier note

Encrypted MFA configuration is stored in Firestore, so enrollment and login
verification still consume a small number of Firestore operations. Firebase
Identity Platform MFA is not required, but an exhausted Firestore quota will
temporarily prevent MFA verification.
